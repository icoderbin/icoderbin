AndroidNative异常捕获原理和xcrash源码分析

更新于2022-07-20|性能监控与优化AndroidNative异常捕获原理和xcrash源码分析

Android平台中，Native crash也是占崩溃的大头，下面介绍如何捕获到Native崩溃信息。

预备知识

用户态和内核态

用户态和内核态是操作系统的两种运行状态。

  • 内核态：处于内核态的 CPU 可以访问任意的数据，包括外围设备，比如网卡、硬盘等，处于内核态的 CPU 可以从一个程序切换到另外一个程序，并且占用 CPU 不会发生抢占情况，一般处于特权级 0 的状态我们称之为内核态。
  • 用户态：处于用户态的 CPU 只能访问受限资源，不能直接访问内存等硬件设备，不能直接访问内存等硬件设备，必须通过「系统调用」陷入到内核中，才能访问这些特权资源。

在CPU指令中，有些指令是非常危险的，所以只能在内核态调用，否则分分钟整个系统就会崩溃。那么什么情况下会由用户态转为内核态呢？

  • 系统调用：这是用户态进程主动要求切换到内核态的一种方式，用户态进程通过系统调用申请使用操作系统提供的服务程序完成工作。
  • 异常：当CPU在执行运行在用户态下的程序时，发生了某些事先不可知的异常，这时会触发由当前运行进程切换到处理此异常的内核相关程序中，也就转到了内核态，比如缺页异常。
  • 外围设备的中断：当外围设备完成用户请求的操作后，会向CPU发出相应的中断信号，这时CPU会暂停执行下一条即将要执行的指令转而去执行与中断信号对应的处理程序，如果先前执行的指令是用户态下的程序，那么这个转换的过程自然也就发生了由用户态到内核态的切换。比如硬盘读写操作完成，系统会切换到硬盘读写的中断处理程序中执行后续操作等。

异常处理机制

函数运行在用户态，当执行过程中发生异常的时候，会切换到内核态。

  • 内核首先接收到异常信号，接收到异常信号后，会把信号放到对应进程的信号队列里。
  • 内核向进程发送一个中断信号，使进程陷入内核态。注意此时信号还在信号队列里，进程还不知道有信号到来。
  • 当进程从内核态变为用户态或者从睡眠状态变为唤醒状态时，会对信号进行检测，此时检测到异常信号。
  • 进程返回到用户态执行信号处理函数。
  • 信号处理函数执行完成后，还需要返回内核态，检查是否还有其它信号未处理。如果所有信号都处理完成，就会将内核栈恢复（从用户栈的备份拷贝回来），同时恢复指令寄存器（eip）将其指向中断前的运行位置，最后回到用户态继续执行进程。

native crash捕获

从上面的分析可知，如果想要捕获native crash，需要对异常信号量进行监听，Android平台中通过sigaction来监听信号处理。

该函数可以用来查询或者设置信号的处理方式，一共有三个参数：

  • signum：信号编码，即要捕获的信号类型，可以是除SIGKILL及SIGSTOP外的任何一个特定有效的信号，如果为这两个信号定义自己的处理函数，将导致信号安装错误。
  • act：新的信号处理方式，该参数指向结构体sigaction的一个实例的指针，该实例指定了对特定信号的处理，如果设置为空，进程会执行默认处理，即使用sa_handler作为信号处理函数。
  • oldact：与act类似，如果不为NULL，则会输出先前信号的处理方式。

信号编码

如下所示为常见的信号编码

信号量 Value 描述 例子
SIGILL 4 非法指令 损坏的可执行文件或者代码区损坏
SIGABRT 6 进程发现错误或者调用了abort() 很多 C 的库函数，如果发现异常会调用 abort()，如 Strlen
SIGBUS 7 不存在的物理地址，硬件有误 更多的是因为硬件或者系统引起的
SIGFPE 8 浮点数运算错误 如除 0 操作，余 0，整型溢出
SIGSEGV 11 段地址错误 空指针，访问不存在的地址空间，访问内核区，写只读空间。栈溢出，数组越界，野指针
SIGPIPE 13 管道错误，往没有 reader 得管道中写 Linux 中的 socket，如果断掉了继续写。signal(SIGPIPE, SIG_IGN)

信号处理方式

sigaction函数的后两个参数表示新的和旧的信号处理方式，调用以后，新的信号处理方式通过act传入，旧的处理方式则会被赋值到oldact以便后续恢复。

信号处理方式为sigaction的数据结构，如下为该数据结构的详细字段：

  • 首先是一个匿名的联合体，包含两个成员sa_handler和sa_sigaction，都代表着信号处理函数。具体使用哪个取决于具体的信号处理需求。使用联合体节省了内存空间，因为同一时间只能使用其中的一个成员。使用那个作为信号处理函数，由sa_flags 决定。
    • sighandler_t sa_handler：类型为 sighandler_t，这是一个函数指针类型，通常表示一个简单的信号处理函数，签名为void handler(int)。
    • void (*sa_sigaction)(int, struct siginfo*, void*)：这是一个更复杂的信号处理函数指针，签名为 void handler(int, struct siginfo*, void*)，用于接收更详细的信号信息。
  • sa_mask 是一个信号集，用于指定在信号处理程序执行期间要屏蔽的信号集合。即在处理该信号时，暂时将sa_mask 指定的信号集搁置，避免嵌套的信号处理程序干扰当前处理程序。
  • sa_flags 用于指定信号处理的行为。
    • SA_RESTART：使被该信号中断的系统调用自动重启。
    • SA_SIGINFO：使用 sa_sigaction 而不是 sa_handler 作为信号处理程序。
    • SA_ONSTACK 是一个信号处理标志，用于指定信号处理程序应该在替代信号堆栈上运行。这个标志通常用于确保信号处理程序在一个独立的、预先分配的堆栈上执行，以避免在处理信号时可能发生的栈溢出问题。
  • sa_restorer是一个函数指针，指向信号处理程序返回后调用的恢复函数。在现代系统中通常不再使用，一般设为 NULL。

信号处理堆栈

native crash有一种堆栈溢出的错误，即分配的堆栈满了，比如说有太多的递归调用就有可能出现这种状况，堆栈满了以后发射SIGSEGV 的中断信号。如果我们自定义的信号处理函数在原有的堆栈上处理，那么原有堆栈满了会导致再次发射SIGSEGV 的中断信号，因此会导致死循环。

所以我们可以使用sigaltstack函数为进程创建一个备用堆栈，供信号处理函数使用。

传入的参数均为stack_t指针类型。

  • __new_signal_stack 是传入新的可替换信号栈。
  • __old_signal_stack 是将旧的可替换信号栈存储在该指针上。

下面是stack_t的数据结构：

  • ss_sp 指向分配的备用堆栈的地址。
  • ss_size 表示分配的地址空间大小。
  • ss_flags 用于指定堆栈的状态和行为。要想创建一个新的可替换信号栈，ss_flags必须设置为0。

xcrash 捕获信号

如上通过sigaltstack函数设置了备用堆栈。然后使用sigfillset将 act.sa_mask 中的所有信号设置为阻塞状态。然后通过循环调用设置每个信号的信号处理函数，这样，信号就成功监听上了。

信号处理函数

信号处理函数是void (*sa_sigaction)(int, struct siginfo*, void*)类型的函数。

第一个参数 int表示信号编码，就是上面注册的那些信号。

第二个参数是信号的详细信息，下面是siginfo的数据结构。

  • si_signo 信号编号，即接收到的信号类型，例如 SIGINT、SIGSEGV 等。
  • si_errno 与信号关联的错误码，表示信号发生时的错误条件。
  • si_code 信号代码，描述信号的具体来源和类型。例如，区分是用户发送的信号还是内核生成的信号。

第三个参数为上下文信息，是一个指向上下文信息的指针，通常用于保存信号处理时的处理器状态和堆栈信息，它是一个ucontext_t 的指针类型。在不同CPU架构中，ucontex_t的数据结构是不一样的。

获取寄存器信息

xcrash 在获取ucontext_t上的参数是，使用了预定义宏，不同的架构通过不同的方式获取。

  • __arm__ 对应32 位 ARM
  • __aarch64__ 对应 64 位 ARM
  • __i386__ 对应32 位 X86
  • __x86_64__ 对应64 位 X86

x86架构

在x86芯片上ucontext_t的数据结构是这样的：

  • uc_link: 指向在信号处理程序返回后将恢复的上下文。如果为 NULL，则信号处理程序返回后将直接返回到内核。
  • uc_sigmask: 信号掩码，指定在信号处理程序运行期间被阻塞的信号集。
  • uc_stack: 用于指定信号处理程序运行时的堆栈。
  • uc_mcontext: 处理器特定的上下文信息，包含寄存器状态。

下面是寄存器的解释：

  • EAX: 累加寄存器，常用于算术运算。
  • EBX: 基址寄存器，常用于存储内存地址。
  • ECX: 计数寄存器，常用于循环操作。
  • EDX: 数据寄存器，常用于输入/输出和算术运算。
  • EDI: 目的索引寄存器，常用于字符串操作。
  • ESI: 源索引寄存器，常用于字符串操作。
  • EBP: 基指针寄存器，通常用于指向函数调用的堆栈帧的基址。
  • ESP: 堆栈指针寄存器，指向当前堆栈的顶部。
  • EIP: 指令指针寄存器，保存当前正在执行的指令的地址。

x86_64 寄存器

下面是x86_64的寄存器信息。

  • RAX: 累加寄存器，常用于算术运算。
  • RBX: 基址寄存器，常用于存储内存地址。
  • RCX: 计数寄存器，常用于循环操作。
  • RDX: 数据寄存器，常用于输入/输出和算术运算。
  • R8-R15: 通用寄存器。
  • RDI: 目的索引寄存器，常用于函数参数传递。
  • RSI: 源索引寄存器，常用于函数参数传递。
  • RBP: 基指针寄存器，通常用于指向函数调用的堆栈帧的基址。
  • RSP: 堆栈指针寄存器，指向当前堆栈的顶部。
  • RIP: 指令指针寄存器，保存当前正在执行的指令的地址。

5.0 以上 libbacktrace.so

xcrash使用的是libunwind，但还有一种使用 libbacktrace的方案。libbacktrace 其实也是用 libunwind 实现的。为了绕开 Android N 以后的 classloader namespace 限制，可以用 ndk_dlopen 来加载 libbacktrace.so。
