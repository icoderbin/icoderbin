xhook源码分析之基本使用

更新于2022-08-11|开源库xhook源码分析之基本使用

xhook 是基于PLT的hook实现，下面对其如何使用，原理如何进行详细的学习。

编译so

下载到源码后，需要将xhook编译成so文件，再在Android工程中接入。xhook提供了编译脚本./build_libs.sh。编译脚本如下

在Android开发中，可以使用 ndk-build 或 CMake 来构建本地代码。两者都有各自的优点和使用场景。

  • ndk-build 是Android NDK提供的传统构建系统，使用 Android.mk 和 Application.mk 文件来定义构建规则。ndk-build更加简单直观，适合较小的项目或已经使用了很多年的项目。
  • CMake 是一个跨平台的构建系统，更加灵活和强大，适合大型项目或需要跨平台支持的项目。

在xhook中使用了传统的ndk-buid的工具，使用ndk-buid构建时，需要将ndk构建工具添加到环境变量中，位置在/~/Library/Android/sdk/ndk-bundle中，如果ndk-bundle中的版本对一些操作不兼容，同样可以在~/Library/Android/sdk/ndk/<version>中找到。

ndk-build

ndk-build 的构建依赖两个文件：Android.mk 和 Application.mk。

  • Android.mk 文件用于定义项目的模块和源文件。它指定如何构建本地代码库.
  • Application.mk 文件用于配置应用程序的构建选项和目标平台。这些设置适用于所有模块。

Android.mk

如下所示为Android.mk的文件内容。

  • LOCAL_PATH: 指定当前Android.mk文件所在的目录。通常使用$(call my-dir)宏来获取这个路径。
  • include $(CLEAR_VARS): 清除之前定义的所有局部变量，确保变量不会受到上一个模块的影响。
  • LOCAL_MODULE: 定义模块的名称。这个名称将用于生成的共享库，上面就会生成libxhook.so。
  • LOCAL_SRC_FILES: 列出模块的源文件，可以是相对路径或绝对路径。
  • LOCAL_C_INCLUDES: 用于指定要包含的头文件目录路径。在构建本地代码时，编译器会搜索这些目录，以找到包含文件（header files）。
  • LOCAL_CFLAGS: 用于向编译器传递额外的编译选项。你可以使用它来设置预处理器宏、优化级别、警告控制等。
    • Wall: 启用所有常见的编译器警告。
    • Wextra: 启用额外的警告，通常是一些更严格的代码检查。
    • Werror: 将所有警告视为错误。这意味着任何警告都会使编译失败，确保代码没有潜在问题。
    • fvisibility=hidden: 设置默认的符号可见性为隐藏，这可以减少共享库的符号表大小并提高性能和安全性。
  • LOCAL_CONLYFLAGS: 用于指定只应用于 C 语言文件的编译选项。它类似于 LOCAL_CFLAGS，但是只影响被编译为 C 语言的源文件，而不包括 C++ 文件。
  • LOCAL_LDLIBS: 是一个用于指定链接本地库的变量。它通常用于告诉链接器将特定的本地库链接到你的共享库（.so文件）中。-llog 是链接器标志，用于链接 Android 系统提供的 liblog.so 库。
  • include $(BUILD_SHARED_LIBRARY): 用于告诉 Android NDK 构建系统生成一个共享库（.so 文件）。

Application.mk

Application.mk 用于设置影响整个应用程序所有本地代码构建的全局选项。该文件允许你指定如下内容：

  • APP_ABI参数: 指定要构建的目标 ABI（Application Binary Interface）。ABI 决定了本地代码将如何与硬件和操作系统进行交互，例如 armeabi-v7a, arm64-v8a, x86, x86_64 等。
  • APP_PLATFORM平台版本: 指定 Android 平台的最低支持版本，例如 android-21、android-28 等。
  • APP_STL类型STL（标准模板库）: 指定 C++ 应该使用的 STL 类型，如 c++_shared（共享版的 GNU STL）或 c++_static（静态版的 GNU STL）等。
  • NDK_TOOLCHAIN_VERSION 编译器版本: 指定要使用的编译器版本，如 clang 或 gcc。

可以看到，第四个调用添加了注释，如果使用-O0方式编译的话，hook会失败。这是怎么回事呢？

GCC提供了多个优化级别选项，用于控制编译器在生成机器代码时对代码进行优化的程度。这些选项通常以 -O 开头，后面跟着数字或字母表示不同的优化级别。以下是常见的 GCC 编译器优化级别选项：

  • -O0: 无优化,编译器只进行基本的代码生成，不做任何优化。通常在开发和调试阶段使用，生成的代码结构与源代码几乎一致，便于理解和调试。
  • -O1: 基本优化。启用了一些基本的优化，如优化不必要的代码。在需要稍微提高性能但仍希望保持编译速度较快的情况下使用。
  • -O2: 中级优化。在 -O1 的基础上增加了更多的优化，如更强的循环优化和内联函数。发布版本时常用，可以提升程序的执行效率，但编译时间可能比 -O0 和 -O1 长。
  • -O3: 高级优化。比 -O2 更进一步的优化，可能包括更激进的指令调度和优化策略。对性能要求较高的程序，尽可能地提升程序的执行速度，但可能会增加编译时间和代码大小。

除了基本的 -O 系列选项外，GCC 还提供了一些更细粒度的优化选项，如：

  • -Os：优化代码大小，尽量减少生成的机器码大小，以节省内存空间。
  • -Ofast：启用了 -O3 级别的优化，并允许非安全数学优化，可能导致数学计算的精度损失。
  • -Og：专为调试目的设计的优化级别，保证了生成的代码能够方便调试，但效率略低于 -O1。

因此：

  • 开发阶段：一般建议使用 -O0 或 -Og，这样生成的代码结构更接近源代码，便于调试和定位问题。
  • 发布阶段：根据程序的性能需求选择合适的优化级别，通常 -O2 或 -O3 可以提升程序的执行效率。

API 使用

下面是xhook的api使用，直接搬运github给的说明了，比较详细。(xHook使用说明)[https://github.com/iqiyi/xHook/blob/master/README.zh-CN.md]

注册hook

  • 在当前进程的内存空间中，在每一个符合正则表达式 pathname_regex_str 的已加载ELF中，每一个调用 symbol 的 PLT 入口点的地址值都将给替换成 new_func。之前的 PLT 入口点的地址值将被保存在 old_func 中。
  • new_func 必须具有和原函数同样的函数声明。
  • 成功返回 0，失败返回 非0。
  • pathname_regex_str 只支持 POSIX BRE (Basic Regular Expression) 定义的正则表达式语法。

忽略部分 hook 信息

  • 根据 pathname_regex_str 和 symbol，从已经通过 xhook_register 注册的 hook 信息中，忽略一部分 hook 信息。如果 symbol 为 NULL，xhook 将忽略所有路径名符合正则表达式 pathname_regex_str 的 ELF。
  • 成功返回 0，失败返回 非0。
  • pathname_regex_str 只支持 POSIX BRE 定义的正则表达式语法。

执行 hook

  • 根据前面注册的 hook 信息，执行真正的 hook 操作。
  • 给 async 参数传 1 表示执行异步的 hook 操作，传 0 表示执行同步的 hook 操作。
  • 成功返回 0，失败返回 非0。
  • xhook 在内部维护了一个全局的缓存，用于保存最后一次从 /proc/self/maps 读取到的 ELF 加载信息。每次一调用 xhook_refresh 函数，这个缓存都将被更新。xhook 使用这个缓存来判断哪些 ELF 是这次新被加载到内存中的。我们每次只需要针对这些新加载的 ELF 做 hook 就可以了。

清除缓存

  • 清除 xhook 的缓存，重置所有的全局标示。
  • 如果你确定你需要的所有 PLT 入口点都已经被替换了，你可以调用这个函数来释放和节省一些内存空间。

启用/禁用 调试信息

  • 给 flag 参数传 1 表示启用调试信息，传 0 表示禁用调试信息。 (默认为：禁用)
  • 调试信息将被输出到 logcat，对应的 TAG 为：xhook。

启用/禁用 SFP (段错误保护)

  • 给 flag 参数传 1 表示启用 SFP，传 0 表示禁用 SFP。 (默认为：启用)
  • xhook 并不是一个常规的业务层的动态库。在 xhook 中，我们不得不直接计算一些内存指针的值。在一些极端的情况和环境下，读或者写这些指针指向的内存会发生段错误。根据我们的测试，xhook 的行为将导致 APP 崩溃率增加 “一千万分之一” (0.0000001)。（具体崩溃率可能会增加多少，也和你想要 hook 的库和符号有关）。最终，我们不得不使用某些方法来防止这些无害的崩溃。我们叫它SFP (段错误保护)，它是由这些调用和值组成的：sigaction()， SIGSEGV， siglongjmp() 和 sigsetjmp()。
  • 在 release 版本的 APP 中，你应该始终启用 SFP，这能防止你的 APP 因为 xhook 而崩溃。在 debug 版本的 APP 中，你应该始终禁用 SFP，这样你就不会丢失那些一般性的编码失误导致的段错误，这些段错误是应该被修复的。
